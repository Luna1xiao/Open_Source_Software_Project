## Goal
Add a Chinese usage guide for `backend/db` so other Mercury team members can call storage APIs without reading SQL or long implementation notes.

## Approach
Document the rule that non-db modules should not write SQL directly, then provide practical examples for Feed, Article, Cleaner, TagAgent, Agent, Provider settings, App config, Usage, Search, Backup, and tests.

## Decisions
Keep examples focused on public imports from `db` and avoid implementation details. Mention that OPML and Markdown export should be built on top of repository APIs rather than placed directly in the storage layer.

## Surprises
The storage API surface is now broad enough that a short module README is more useful for teammates than the longer implementation guide.

## Follow-ups
Update this README whenever repository APIs are renamed or route-level contracts are added.
